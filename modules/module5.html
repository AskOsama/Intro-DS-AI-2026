Anatomy of a composed agent

The four days behind you each delivered one piece of a working agent. Today is where they snap together. The simplest useful definition:

Agent = an LLM (Day 4) + tools (Day 4 building blocks) + memory (Day 2) + a safety wrapper (Day 3) + a loop that runs them (Day 2).

That's it. Every agent your team will build, buy, or be sold — from the simplest workflow bot to the most ambitious computer-use agent — is a composition of those five ingredients. When you understand which ingredient does what, you can read any agent proposal as a parts list, not magic.

The five parts, mapped to the days

Part What it does Where you met it
The LLM The "brain". Reads context, proposes the next action, drafts language. Hallucinates if not constrained. Day 4 — generative AI segment
Tools What the LLM can do. Each tool is a capability the agent can call. Day 4 building blocks (see four common ones below)
Memory / context What the agent remembers across turns. Short-term in the context window; longer-term in files, databases, MCP servers. Day 2 (context window) + Day 3 (where memory physically lives)
Safety wrapper Autonomy level, action approval, action log, kill-switch, identity. Day 2 (responsibility checklist) + Day 3 (agent-specific security)
The loop The runtime that drives Read → Plan → Act → Observe → Repeat. Day 2 (the agent loop)

Four common tools an agent can have

The "tools" slot is where your Day-4 building blocks plug in. Four that recur in real agents:

  1. An optimization tool. The agent calls it with constraints and an objective; it returns the best feasible plan. Use case: "schedule six inspectors across twelve sites this week, minimize travel time, no inspector visits the same site twice." Day-4 root: search & optimization.
  2. A database / KBS tool. The agent looks up facts (what does the policy say about X?) or applies rules (does this case meet the eligibility conditions?). Use case: a procurement agent that checks each proposed clause against your data-classification rule book. Day-4 root: knowledge-based systems.
  3. A reasoning / ML tool. A specialised model the agent can call when its general LLM isn't precise enough — e.g., a fraud-classifier, a forecast model, a sentiment scorer. Use case: an inbound-letters agent that asks a fine-tuned classifier "is this a compliance complaint?" before drafting a response. Day-4 root: machine learning.
  4. A safe action tool. The agent's fingers — what it can change in the world. Wrapped so every call is logged, every above-threshold call is gated by human approval, and every action below the autonomy line is allowed to happen automatically. Use case: "send the drafted reply" is a safe action; "transfer money" almost never is. Days 2 and 3 between them.

Worked example: a Procurement Review Agent

How the parts compose on a real task

The job. Your team receives ~30 vendor proposals per month for AI tooling. Each one needs a rapid review against three things: NDMO classification fit, total cost of ownership compared to the current incumbent, and a written summary for the director.

The agent.

  • LLM: Claude Opus (hosted) or a Qwen-class model (on-prem) — chosen based on the highest classification level any proposal will contain.
  • KBS tool: the NDMO classification rules + your organisation's PDPL playbook, exposed as a local MCP server. The agent asks: "Given this proposal mentions [X], what classification level applies, and what tools are permitted?"
  • Optimization tool: a small TCO solver. The agent feeds in proposal numbers vs. incumbent; the solver returns the Pareto front of cost/capability trade-offs and identifies the dominated proposals.
  • Reasoning / ML tool: a small classifier you trained once on past procurement decisions, returning "approve / refer / reject" with a confidence. Used as a sanity check on the LLM's draft recommendation.
  • Memory: the proposal text + every previous review of the same vendor, fetched on the fly via an MCP server pointed at your procurement archive.
  • Safety wrapper: autonomy level supervised — the agent drafts everything, but a human procurement officer must click "Send" before the recommendation reaches the director. Action log records every tool call (which classification rule fired, which TCO assumption used, which classifier confidence). Kill-switch: disabling the MCP server stops all activity within seconds.
  • The loop: Read the proposal → Plan (which tools to call in what order) → Act (call the tools, draft the summary) → Observe (results, classifier disagreement, missing fields) → Repeat until the draft is complete and ready for human review.

What you, the leader, do. You don't write any of the code. You do specify the proposal that says which LLM, which tools, which autonomy level, and which classification level the system handles. That specification is a one-page document — the same one you're about to draft below.

The whole point

The fields in the §template section that follows are not arbitrary — each one names one of the parts above. AI Approach picks the LLM. Data Needed determines the memory. Classification & Sovereignty chooses where the LLM and tools physically run. Autonomy Level sets the safety wrapper's strictness. Privacy & Risk names the threats that wrapper has to defend against. Success Metric tells the loop when it's done. You are spec-ing an agent. The template is the schema.

The proposal template

One page. Seven fields. Aim for one or two paragraphs per field.

1. Problem What's slow, error-prone, or repetitive in your work? Be specific. Numbers help: "We process 2,000 incoming letters per month and it takes our team an average of 18 hours/week."
2. AI Approach Which kind of AI from this week? Chatbot drafting? ML prediction? Document classification? KBS over a policy book? Generative? Agent (Claude Code, OpenCode, custom MCP)? Be precise — "use AI" is not an answer.
3. Data Needed What data does the approach require? What exists today (and where)? What doesn't exist? If labelled training data is needed, who would label it?
4. Classification & Sovereignty What NDMO classification level does this data carry? Does the proposed approach (hosted, local, hybrid) keep the data inside the right boundary under PDPL? If using MCP servers, where do they run?
5. Autonomy Level (New for 2026.) If this involves an agent: suggestion-only, per-action approval, bounded autonomous, or fully autonomous? What's the kill-switch and the action-log? Tied to Day 2's autonomy spectrum and Day 3's agent procurement questions.
6. Privacy & Risk What could go wrong? Who's harmed if it does? Hallucination? Re-identification? Prompt injection (if agent)? Wrong action by the agent? What's your mitigation for each?
7. Success Metric How would you know it worked? Time saved? Accuracy improved? Cost reduced? Be quantitative. "Improved" is not measurable.

Common mistakes to avoid

  • Generic "AI" in field 2. If you can't name which kind — chatbot, ML, KBS, agent — you don't have a proposal yet.
  • Skipping field 4. If you can't classify the data, you don't know whether your proposal is legal.
  • Skipping field 5. If the proposal involves an agent and you haven't named the autonomy level, you've handed the decision to a vendor.
  • "Improved" or "better" in field 7. If the metric isn't a number, you can't tell whether it worked.
  • Solving someone else's problem. If you don't own or influence the process, the proposal goes nowhere.

A worked example

Imagine an HR department that processes incoming employment-verification letters. Here is what a strong one-page proposal might look like, to give you a target.

Example: Automated employment-verification triage

1. Problem. Our HR team receives ~600 employment-verification requests per month from banks, embassies, and other employers. Each one currently takes an analyst about 15 minutes to look up, draft, and send — ~150 hours per month, ~1.5 FTE. Most are routine; ~5% are exceptions that need human judgment.

2. AI Approach. A two-stage system. Stage 1: a document-classification model (supervised ML, Day 4) reads the incoming letter and classifies it as routine or exception, plus extracts the requester, employee name, and requested fields. Stage 2: for routine cases, a template-driven generator (LLM with strict prompt and verification, Day 2 methodology) drafts the response for analyst approval. Exceptions are flagged for human handling.

3. Data Needed. 18 months of past requests + responses (~10,000 documents) exist in a shared drive. Labelling routine vs. exception requires ~2 weeks of analyst time (already budgeted). No new data collection required.

4. Classification & Sovereignty. Letters contain employee personal data → Confidential under NDMO. Hosted Tier-1 LLMs (e.g., Claude.ai) ruled out per PDPL cross-border restrictions. Recommend on-premises deployment of an open-source model (e.g., Qwen-2.5 or Gemma 4) accessed via OpenCode + a custom on-prem MCP server that fronts the HR database. Day 3 hosted/local trade-off accepted: slightly lower quality, full sovereignty.

5. Autonomy Level. Supervised in pilot — the agent drafts but an analyst reviews and approves every routine response before send. After 90 days, evaluate moving to bounded autonomous for the highest-confidence subset (response auto-sent if confidence > 95% AND date cross-check passes; everything else routes to human). Kill-switch: disabling the MCP server stops all agent activity within seconds. Action log: every tool call logged with timestamp, agent identity, requester, and outcome.

6. Privacy & Risk. (a) Hallucination risk: model invents employment dates. Mitigation: all generated dates cross-checked against the source HR database before send; analyst signs off on every routine response in pilot phase. (b) Misclassification risk: exception treated as routine. Mitigation: conservative threshold; below-confidence cases default to human. (c) Prompt-injection risk: a malicious requester embeds instructions in the letter body. Mitigation: the agent's MCP tools are restricted to read-only HR-database queries and template-driven response composition; no tool can change records or send mail directly. (d) Bias risk: certain employee categories systematically misclassified. Mitigation: audit by employee category quarterly.

7. Success Metric. Reduce average analyst time per request from 15 to ≤ 5 minutes (66% reduction) at ≥ 99% accuracy on routine cases over a 90-day pilot. If achieved, FTE saving = ~1.0 (60% of current 1.5).

Notice what the example does: every field is specific. Numbers are real. The risk section names three concrete risks and one mitigation each. The success metric is a number with a deadline. That's the bar.

Putting the template to use

The template is a checklist; the work is your judgment. A pragmatic order to fill it in:

  1. Pick a real process you own or influence. Not a theoretical one. The test of whether you can use what you learned this week is whether the process is specific enough to put a number on its current pain.
  2. Sketch the agent first, fields second. Use the §anatomy section above. Decide which LLM tier (hosted/local), which tools (optimization, KBS, classifier, action), what memory it needs, what safety wrapper it lives inside. Then the template fields almost write themselves.
  3. One paragraph per field. Two if needed. The discipline is the work, not the prose. If a field needs three paragraphs you don't yet understand the field.
  4. Show the draft to one trusted peer outside this material. If they can repeat back what you're proposing without referring to your notes, you're ready. If not, the proposal is not ready — it's the proposal that needs sharpening, not your peer.

Sharpening questions to put to your own draft

Use these as a self-review pass

  • "What single number would tell me this is working?"
  • "Where does this data live today, physically?"
  • "What's the worst-case mistake the agent could make? Who would notice, and how soon?"
  • "Is this the kind of AI from Day 4 I actually need, or a different one?"
  • "Who has to say yes for this to happen? Have I made it easy for them to say yes?"

Common patterns — what most proposals turn into

Across cohorts, the proposals that come back successful tend to fall into one of four buckets. Naming yours up front sharpens it:

  • Automation. Repetitive document tasks — verification letters, form processing, simple Q&A. Usually LLM-driven, often Confidential data, on-premises preferred. Watch for: the 5% of cases that aren't routine and need an escape hatch.
  • Document analysis. Triage, classification, extraction from incoming documents. Often a mix of CV (for scans) and NLP (for text). Watch for: sample bias if your training documents come from one source.
  • Prediction. Demand forecasting, no-show estimation, risk scoring. Classical ML — often the right tool, often less hyped than generative AI. Watch for: drift; the world that produced your training data is not the world your prediction lives in.
  • Decision support. Eligibility checking, policy compliance, regulatory reasoning. Often KBS — sometimes hybridised with ML for the fuzzy parts. Watch for: the failure mode is "looks correct but applies the wrong rule" — only audit catches it.

When you're ready to take it to the team

A proposal is ready when (a) the seven fields are filled with specifics not generalities, (b) you can explain it in two minutes without notes, (c) one trusted peer can repeat it back, and (d) you know which person's "yes" makes it real. If any of those four are missing, iterate before bringing it to your IT team or director — their time is the expensive part.

Closing — what you can do now

You came in able to use AI as a consumer. You leave able to direct it deliberately. Concretely:

  • You can evaluate AI proposals from your technical teams. You know which questions the proposal must answer and which silences are red flags.
  • You can ask the right procurement questions. Where the data goes, who has access to logs, what the retention policy is, what runs on-premises, what the cost difference is.
  • You can identify opportunities in your own work. The capstone you just wrote is one. There will be others as the methodology becomes a habit.
  • You can make data-classification-aware decisions about which AI tools to deploy where. Public, Confidential, Secret, Top Secret — each level constrains the toolbox, and you know how.
  • You can tell good AI from bad AI across the four perspectives, the eight subfields, and the eras. You know when "AI" is hype and when it's substance.
  • You can sketch what a composed agent looks like for your problem: which LLM, which tools (optimization, KBS, classifier, action), what memory it needs, what safety wrapper it sits inside. You're not just naming the type of AI — you're spec-ing the parts list.

The one-line summary

Direct AI deliberately. Match the tool to the data classification. Ask the trade-off question every time. Verify everything load-bearing. Repeat.
