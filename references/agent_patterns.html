The augmented LLM — the building block

Every pattern below assumes one foundational thing: an augmented LLM — in Anthropic's words, "an LLM enhanced with augmentations such as retrieval, tools, and memory." The model can autonomously generate search queries, pick which tool to call, and decide what information is worth keeping in working memory.

Without these augmentations, an LLM is a smart but isolated text processor. With them, it becomes the brain of a system that can reach into the world. Every pattern that follows is just a different way of composing augmented LLMs together.

Worked example — the augmented LLM in your daily life

When you ask Claude or ChatGPT to summarise an attached PDF, three augmentations fire at once: retrieval (the file is loaded into context), tools (a code-runner parses the PDF), and memory (the model keeps track of what you've already asked). The "magic" is the combination, not any single ingredient.

Workflow vs. agent — the key distinction

The single most important sentence in the essay is the one that draws this line:

Workflows are systems where LLMs and tools are orchestrated through predefined code paths. Agents are systems where LLMs dynamically direct their own processes and tool usage, maintaining control over how they accomplish tasks.

— Anthropic, Building Effective Agents (2024)

In a workflow, a human engineer wrote down the steps and the LLM fills in the blanks. In an agent, the LLM chooses the steps. The five patterns on this page are workflows. The last short section is about agents proper. The reason the distinction matters: workflows are predictable and easy to audit; agents are powerful and hard to bound. Most production systems today are workflows pretending to be agents in marketing copy.

Pattern 1 — Prompt chaining

Definition. Decompose a task into sequential steps, where each LLM call processes the output of the previous one. A simple gate (a regex check, a classifier, a human review) can sit between steps to catch failures before they propagate. 

   ┌────────┐    ┌────────┐    ┌────────┐    ┌─────────┐
 → │  LLM 1 │ →  │  Gate  │ →  │  LLM 2 │ →  │  Output │
   └────────┘    └────────┘    └────────┘    └─────────┘

When to use. Whenever the task is naturally a multi-step pipeline and each step is more reliable in isolation than as one big prompt. The classic example: write a draft, then translate it, then check the translation against the original.

Public-sector example

A ministry receives 200 complaint emails a day in mixed Arabic and English. A prompt-chain workflow: (1) detect language, (2) summarise the issue in 30 words, (3) classify the department it belongs to, (4) draft a one-line acknowledgement in the sender's language. A human approves before send. Each step is small enough that a less-capable, cheaper model can run it.

Pattern 2 — Routing

Definition. Classify the input first, then dispatch it to the prompt (or model, or specialised pipeline) that handles that class best. The router itself is usually a small, cheap LLM call. 

                            ┌──────► [ Refund handler ]
   ┌─────────┐              │
 → │ Router  │ ─ classify ─►├──────► [ Technical-support handler ]
   └─────────┘              │
                            └──────► [ Account-closure handler ]

When to use. When inputs come in distinct flavours and a single "do-everything" prompt would be muddled or wasteful. Routing also lets you mix model tiers: use a cheap model for easy questions and reserve a frontier model for the hard ones.

Public-sector example

A national tax helpdesk: the router sees the question, decides whether it's about individual income, corporate VAT, or a procedural refund, and routes to the prompt that has the right form numbers and references baked in. The handler prompt for VAT never has to know what an individual-income deduction is — cleaner prompts, fewer mistakes.

Pattern 3 — Parallelization

Definition. Run multiple LLM calls at the same time and combine their outputs. Two flavours: sectioning (split a job into independent pieces) and voting (run the same prompt several times and aggregate, e.g. majority-vote a classification). 

                  ┌──► LLM (red-team) ──┐
                  │                     │
   ┌────────┐ ───┼──► LLM (legal) ─────┼──► [ Aggregator ] → Final review
   │ Input  │    │                     │
   └────────┘ ───┼──► LLM (security) ──┘

When to use. When you want multiple lenses on the same input (each independent, no shared context), or when a single answer benefits from the wisdom of crowds. Latency drops because the calls happen in parallel; quality often rises because the lenses catch different failures.

Public-sector example

Reviewing a draft policy document: one LLM call checks for legal issues, another for security implications, a third for plain-language clarity. They run in parallel, and a final aggregator (or a human) consolidates the three reports. None of the three reviewers needs to know about the others' findings.

Pattern 4 — Orchestrator-workers

Definition. A central LLM (the orchestrator) breaks the task into subtasks at runtime, dispatches each to a worker LLM, then synthesises the workers' answers. The difference from parallelization: the orchestrator decides what the subtasks even are, so the structure isn't fixed in code. 

                       ┌─────► Worker A ──┐
                       │                  │
   ┌──────────────┐ ──┼─────► Worker B ──┼──► [ Orchestrator ]
   │ Orchestrator │   │                  │      synthesises
   └──────────────┘ ──┴─────► Worker C ──┘

When to use. For tasks where you don't know in advance how many subtasks exist or what they look like. A research-style query ("compare these 12 vendors on five dimensions") is a natural fit; a simple classification is not.

Pattern 5 — Evaluator-optimizer

Definition. One LLM produces a draft. A second LLM (the evaluator) reads it and writes feedback. The first LLM revises. Repeat until the evaluator is satisfied or a budget is exhausted. 

   ┌───────────┐    draft     ┌─────────────┐
 → │ Generator │ ───────────► │  Evaluator  │
   └───────────┘              └──────┬──────┘
        ▲                            │
        │           feedback         │
        └────────────────────────────┘

When to use. When quality is the bottleneck and you have a clear way to judge a draft. Translation that needs to preserve nuance, code that needs to pass tests, a policy memo that needs to hit a tone — all benefit. If you can't write down what "good" looks like, you can't build the evaluator.

When to use an agent (and when not to)

The five patterns above are workflows: a human engineer chose the structure; the LLM fills it in. Agents are different — the LLM itself decides what to do next, in what order, with what tools.

Use agents for "open-ended problems where it's difficult or impossible to predict the required number of steps." But "you should consider adding complexity only when it demonstrably improves outcomes."

— Anthropic, Building Effective Agents

Translated for a leader: most of the systems your vendors will pitch as "agents" are workflows wearing an agent costume. That's usually a good thing — workflows are auditable and bounded. Reach for a true agent only when the task genuinely cannot be scripted, and budget for the guardrails (tool restrictions, rate limits, human-in-the-loop) that an autonomous loop demands.

The leader's question

When a vendor says "our agent does X", ask: "Is the structure of X decided by your code, or by the model at runtime?" The first answer is a workflow (good, predictable). The second is an agent (powerful, needs strong guardrails). Both are valid — but you should know which one you're buying.
